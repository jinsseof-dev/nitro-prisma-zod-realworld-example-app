바이브 코딩 마이그레이션 튜토리얼을 생성하거나 업데이트합니다. 실제 프로젝트에서 사용된 프롬프트와 결과를 기반으로 다른 개발자들이 쉽게 따라할 수 있는 실전 가이드를 작성합니다.

## 튜토리얼 구조

아래 구조로 `docs/vibe-coding-tutorial.md` 파일을 작성합니다:

```markdown
# 바이브 코딩 마이그레이션 가이드

## 개요

- 바이브 코딩이란?
- 이 가이드의 목적과 대상 독자
- 전제 조건 (Claude Code 설치, 프로젝트 기본 구조)

## Phase 1: AI 컨텍스트 설정

- CLAUDE.md 작성
- 실제 사용 프롬프트와 결과

## Phase 2: 코드 품질 가드레일

- ESLint + Prettier 설정
- Pre-commit Hook (Husky + lint-staged)
- 실제 사용 프롬프트와 결과

## Phase 3: 테스트 안전망

- 유닛 테스트 작성
- 테스트 커버리지 확보
- 실제 사용 프롬프트와 결과

## Phase 4: CI/CD 파이프라인

- GitHub Actions 설정
- 통합 테스트 연동
- 실제 사용 프롬프트와 결과

## Phase 5: 리뷰 & 개선 사이클

- 프로젝트 리뷰 요청
- 이슈 등록 및 관리
- 크리티컬 이슈 해결
- 실제 사용 프롬프트와 결과

## Phase 6: 워크플로우 자동화

- Claude Hook 설정
- 슬래시 커맨드 생성
- 병렬 작업 규칙
- 실제 사용 프롬프트와 결과

## 부록

- 전체 프롬프트 목록 (요약)
- 트러블슈팅 (실제 겪은 문제와 해결법)
- 체크리스트
```

## 데이터 소스

1. **작업 로그**: `~/.claude/projects/<프로젝트명>/` 하위 JSONL 파일에서 사용자 프롬프트 추출
2. **OpenSpec 아카이브**: `openspec/changes/archive/` 하위의 proposal, design, tasks 참조
3. **Deep Interview 스펙**: `.omc/specs/deep-interview-vibe-coding-setup.md` 참조
4. **결과물**: CLAUDE.md, eslint.config.mjs, .prettierrc, .husky/, ci.yml, 테스트 파일들, .claude/ 설정

## 작성 규칙

- 언어: 한국어
- 각 Phase에 **실제 사용된 프롬프트**를 `> 프롬프트` 블록으로 표시
- 프롬프트 다음에 **결과 요약**을 작성 (어떤 파일이 생성/수정되었는지)
- 트러블슈팅 섹션에는 실제 발생한 에러와 해결 과정을 포함
- 각 Phase 끝에 **검증 방법**을 포함 (어떤 명령어로 확인하는지)
- 실무에서 바로 복사-붙여넣기할 수 있는 프롬프트 형태로 작성
- 프로젝트에 종속되지 않도록 일반화된 가이드 + 이 프로젝트의 구체적 예시 병행

## 실행 절차

### 생성 (인자 없이 실행)

1. 위 데이터 소스를 모두 읽습니다.
2. 작업 로그에서 사용자 프롬프트와 결과를 추출합니다.
3. 위 구조에 맞게 `docs/vibe-coding-tutorial.md`를 작성합니다.
4. 생성 완료 후 파일 경로를 보고합니다.

### 업데이트 (인자로 Phase 지정)

1. 기존 `docs/vibe-coding-tutorial.md`를 읽습니다.
2. 지정된 Phase 섹션만 최신 정보로 업데이트합니다.
3. 변경된 내용을 보고합니다.
