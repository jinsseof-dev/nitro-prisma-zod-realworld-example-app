GitHub 이슈를 생성합니다. 사용자의 설명을 기반으로 구조화된 이슈를 작성합니다.

## 이슈 구조

아래 템플릿에 따라 이슈를 작성합니다:

```markdown
## 배경

- 현재 상태와 문제점을 간결하게 설명
- 왜 이 작업이 필요한지 근거 제시

## 개요

- 작업의 목표와 범위
- 주요 작업 영역을 표로 정리

| 영역      | 내용 |
| --------- | ---- |
| **영역1** | 설명 |

### 태스크 목록

- [ ] 태스크 1
- [ ] 태스크 2

## 인수 조건 (Acceptance Criteria)

- [ ] 조건 1
- [ ] 조건 2

## 의존 관계

- 다른 이슈나 작업과의 의존 관계 기술
- 없으면 "이 이슈는 독립적으로 진행 가능합니다." 표기
```

## 실행 절차

### 이슈 생성

1. 사용자 입력에서 이슈 제목과 내용을 파악합니다.
2. 위 템플릿 구조에 맞게 이슈 본문을 작성합니다.
3. `gh issue create` CLI로 이슈를 생성합니다.
4. 라벨이 필요하면 `--label` 옵션을 추가합니다.
5. 생성된 이슈 URL을 사용자에게 보고합니다.

### 이슈 업데이트 (작업 완료 시)

1. 해당 이슈의 **인수 조건**을 읽습니다 (`gh issue view <번호>`).
2. 각 인수 조건을 **코드 증거 기반으로 검증**합니다:
   - 파일:라인 단위로 변경사항 확인 (Grep/Read 사용)
   - 테스트 실행 결과 확인 (`bun test`)
   - 검증 불가능한 조건은 사유와 함께 보고
3. 검증 결과를 이슈 코멘트로 작성합니다:

   ```markdown
   ## 인수 조건 검증 결과

   - [x] **조건 1** — 확인됨 (`파일명:라인` 증거)
   - [x] **조건 2** — 확인됨 (테스트 결과 증거)
   - [ ] **조건 3** — 미충족 (사유)

   ### 테스트 결과

   - 유닛 테스트: N pass, 0 fail
   - CI: 통과/미확인
   ```

4. 모든 인수 조건이 충족되면 이슈를 닫습니다 (`gh issue close <번호>`).
5. 미충족 조건이 있으면 이슈를 열어둔 채 미충족 사항을 보고합니다.

## 규칙

- 리포지토리: `jinsseof-dev/nitro-prisma-zod-realworld-example-app`
- 언어: 한국어 (제목, 본문 모두)
- 태스크 목록은 체크박스(`- [ ]`)로 작성
- 인수 조건은 검증 가능한 형태로 작성
- 의존 관계가 있으면 이슈 번호(`#N`)로 연결
- `gh` CLI를 사용하여 이슈를 생성 (MCP 도구 대신)
- 본문은 HEREDOC(`<<'EOF'`)을 사용하여 전달
- **이슈 업데이트 시 반드시 인수 조건을 코드 증거와 함께 검증**
- 인수 조건 검증 없이 이슈를 닫지 않음
